## Findings

- Anthropic launched Agent Skills on Oct 16, 2025 as "folders that include instructions, scripts, and resources that Claude can load when needed," working across Claude apps, Claude Code (via `~/.claude/skills` or the anthropics/skills plugin marketplace), and the API (which requires the Code Execution Tool beta). [1]
- A skill's core design principle is three-level progressive disclosure: (1) YAML frontmatter `name` + `description` pre-loaded into the system prompt at startup, (2) the full SKILL.md body loaded only when Claude judges it relevant, (3) additional bundled files (e.g. `reference.md`, `forms.md`) read only as needed — so bundled context is "effectively unbounded" for agents with a filesystem and code execution. [2]
- The anthropics/skills repo (created 2025-09, ~162,000 stars by 2026-07) contains example skills, the Agent Skills specification (`./spec`, with a broader standard at agentskills.io), a template, and the source-available document skills (docx/pdf/pptx/xlsx) that power Claude's production document features. [3]
- Anthropic reports (vendor self-reported internal eval) that an orchestrator-worker multi-agent system with Opus 4 lead + Sonnet 4 subagents outperformed single-agent Opus 4 by 90.2% on their internal research eval, and that token usage alone explains 80% of performance variance on BrowseComp — but agents use ~4x and multi-agent systems ~15x the tokens of chat. [4]
- Anthropic's core delegation rule: each subagent task description must include an objective, an output format, guidance on tools/sources, and clear task boundaries — vague instructions like "research the semiconductor shortage" caused duplicated work and gaps. [4]
- Anthropic embedded explicit effort-scaling rules in prompts: simple fact-finding = 1 agent with 3-10 tool calls; direct comparisons = 2-4 subagents with 10-15 calls each; complex research = 10+ subagents with divided responsibilities; plus heuristics to start with short broad queries then narrow, prefer specialized tools over generic ones, and use extended/interleaved thinking as a planning scratchpad. [4]
- Their pipeline includes an explicit citation pass: after the research loop ends, a dedicated CitationAgent processes documents and the draft report to attach source locations to claims; two-level parallelism (lead spawns 3-5 subagents in parallel; each subagent uses 3+ tools in parallel) cut research time by up to 90%; the lead agent saves its plan to external Memory because context past 200K tokens gets truncated. [4]
- "Building effective agents" (Dec 2024) advises finding the simplest solution possible — most successful implementations use "simple, composable patterns" not frameworks — distinguishing predefined workflows (prompt chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer) from autonomous agents, with orchestrator-workers suited to tasks where subtasks can't be predicted in advance. [5]
- For long-horizon research tasks, Anthropic's context-engineering post prescribes three techniques — compaction (summarize and reinitialize the context window, starting with safe tool-result clearing), structured note-taking (agent persists NOTES.md/to-do state outside context and re-reads after resets), and sub-agent architectures — under the principle that context is a finite "attention budget" and agents should use just-in-time retrieval via lightweight identifiers (file paths, links) rather than pre-loading everything. [6]
- LangChain's multi-agent docs (2026) formalize four patterns — Subagents (main agent coordinates subagents as tools; strong context isolation, stateless, one extra call per task), Handoffs, Skills (single agent stays in control, loading specialized prompts on demand), and Router — noting subagents win for parallel multi-domain work (~9K tokens vs ~15K for Skills on a 3-way comparison) while stateful patterns save 40-50% of calls on repeat requests. [7]

## Deep Read Notes

### Source [4]: How we built our multi-agent research system
Key data: 90.2% improvement of multi-agent (Opus 4 lead + Sonnet 4 subagents) over single-agent Opus 4 on internal research eval; token use explains 80% of BrowseComp variance (tool-call count and model choice explain most of the rest, ~95% total); multi-agent = ~15x chat token cost; parallel subagent spawning + parallel tool calls cut research time up to 90%; a rewritten tool description (produced by a self-improving "tool-testing agent") cut task completion time 40%; early failure modes included spawning 50 subagents for simple queries.
Key insight: The whole system is steered by prompts, and the eight prompt-engineering principles are the method: think like your agents (simulate them step-by-step); teach the orchestrator to delegate with objective/format/tools/boundaries per subtask; scale effort to query complexity with explicit numeric rules; treat tool descriptions as a first-class interface with selection heuristics; let Claude rewrite its own failing prompts/tool descriptions; start wide then narrow; use extended thinking (lead) and interleaved thinking (subagents) as controllable scratchpads; parallelize at both the subagent and tool-call level. End-to-end flow: LeadResearcher plans -> saves plan to Memory (200K-token truncation risk) -> spawns subagents -> synthesis loop -> dedicated CitationAgent pass -> final report. Evaluation: start with ~20 real queries immediately; a single LLM judge with one rubric prompt (factual accuracy, citation accuracy, completeness, source quality, tool efficiency; 0.0-1.0 + pass/fail) aligned best with humans. Explicit fit caveat: multi-agent suits breadth-first, parallelizable, high-value tasks; poor fit where agents must share full context or have many inter-dependencies (e.g. most coding).
Useful for: The canonical primary source on orchestrated research pipelines; every methodological ingredient (decomposition rules, effort scaling, citation pass, parallelism) is stated explicitly and quantified.

### Source [2]: Equipping agents for the real world with Agent Skills
Key data: A skill = directory with SKILL.md (required YAML frontmatter: name, description); only name+description of every installed skill sits in the system prompt at startup; the production PDF skill splits form-filling instructions into a separate forms.md so the core stays lean; context-window sequence: metadata -> agent invokes Bash to read SKILL.md -> optionally reads bundled files -> proceeds with task.
Key insight: Skills are the "single-prompt" assembly path: rather than orchestrating multiple agents, you package procedural expertise as progressively-disclosed files that any capable agent (Claude Code, claude.ai, API) loads on demand — the mechanism is deliberately analogous to "a well-organized manual: table of contents, chapters, appendix." Because loading is filesystem-mediated, skill size is effectively unbounded without polluting context, which is exactly the property community deep-research skills exploit (multi-phase pipelines, personas, validation scripts all bundled in one skill folder).
Useful for: Explaining how one well-engineered SKILL.md turns a generalist agent into a research agent, and the progressive-disclosure design rationale.

### Source [8]: 199-biotechnologies/claude-deep-research-skill (community exemplar)
Key data: Single Claude Code skill implementing a 3-8+ phase pipeline (Scope -> Plan -> Retrieve -> Triangulate -> Outline -> Synthesize -> Critique with loop-back -> Refine -> Package); four effort modes (Quick 2-5 min to UltraDeep 20-45 min); 5-10 concurrent searches plus 2-3 sub-agents returning structured evidence objects; disk-persisted `sources.json` that "survives context compaction and continuation agents"; quality gates of 10+ sources and 3+ per major claim; automated `validate_report.py` and `verify_citations.py` (DOI/URL/hallucination checks) in a validate-fix-retry loop (max 3 cycles); multi-persona red-teaming; a "Step 0" that retrieves the current date to prevent stale training-data assumptions.
Key insight: Community skill authors have independently converged on the same ingredients Anthropic's multi-agent post describes — effort scaling by mode, parallel retrieval, sub-agent fan-out, citation verification passes, external memory for compaction survival — but packaged entirely inside one skill folder driving a single Claude Code session, showing the two "assemble it yourself" approaches converging in practice.
Useful for: Concrete evidence of what a state-of-the-art community deep-research skill contains, and the skills-vs-orchestration convergence argument.

## Gaps

- Could not find any independent (non-vendor) benchmark comparing a single-agent-plus-skill setup against an orchestrated multi-agent pipeline on the same research tasks — Anthropic's 90.2% figure is self-reported on an internal eval, and no third-party replication surfaced in this session's searches.

# Orchestrated Multi-Agent Research Pipelines

## Definition

An orchestrator-worker (also called "orchestrator-subagent" or "supervisor-worker") research pipeline is an architecture in which a lead LLM agent — the orchestrator — receives a research query, decomposes it into a set of independent sub-questions, and dispatches each sub-question to a separate worker sub-agent that runs its own tool-calling loop (typically web search plus page-fetch) in its own context window. Workers report condensed findings back to the orchestrator, which repeats delegation as needed, then hands the accumulated evidence to a synthesis (and often a dedicated citation) pass that produces the final report. This is distinct from a single long-running agent that does all planning, searching, and writing itself in one context, and distinct from a hosted black-box product where the internal architecture isn't inspectable — orchestrator-worker pipelines are explicitly *engineered* as multi-agent systems, whether hand-rolled, built on a graph-orchestration framework, or exposed via a coding agent's native subagent tooling.

## How it works

The clearest public description of this pattern is Anthropic's own engineering write-up of the system behind Claude's "Research" feature, published June 13, 2025 [primary] (https://www.anthropic.com/engineering/multi-agent-research-system). Their design has four stages:

1. **Planning**: A `LeadResearcher` agent analyzes the user's query, thinks through a strategy in an "extended thinking" scratchpad, and saves its plan to external memory — necessary because, per Anthropic, once a research task runs long the accumulated context can exceed the 200k-token window, so the plan must survive context truncation rather than living only in-context.
2. **Parallel delegation**: The lead spawns 3–5 subagents simultaneously (more for genuinely broad questions), each given an explicit objective, output format, suggested tools, and clear task boundaries — Anthropic found that vague instructions like "research X" caused agents to duplicate work or leave gaps, so the orchestrator prompt is engineered to over-specify delegation. Anthropic also encodes explicit scaling heuristics into the orchestrator's prompt: simple fact-lookups get 1 agent with 3–10 tool calls; comparisons get 2–4 subagents with 10–15 calls each; broad/complex research gets 10+ subagents with clearly divided responsibilities.
3. **Independent execution**: Each subagent runs its own tool loop against live sources, using interleaved thinking to evaluate result quality, adapt its query, and decide when it has "enough," then compresses its findings into a report back to the lead. Subagents themselves parallelize — each one is encouraged to fire 3+ tool calls concurrently rather than serially, which Anthropic reports cut research time by up to 90% on complex queries versus sequential execution.
4. **Synthesis and citation**: Once the lead judges the evidence sufficient, it exits the research loop and hands the full set of subagent reports to a separate `CitationAgent`, whose sole job is to map specific claims in the synthesized report back to source locations — kept as a distinct final pass rather than folded into synthesis, reportedly to improve citation precision.

The "orchestrator-worker" terminology itself comes from Anthropic's post and has since propagated into the wider agent-framework literature (see LangChain's discussion of the "supervisor pattern," which is functionally the same idea under a different name — https://www.langchain.com/blog/how-and-when-to-build-multi-agent-systems, June 16, 2025 [secondary]).

Other frameworks implement the same mechanics with different vocabulary. LangGraph-based implementations (e.g., LangChain's open-source `open_deep_research`, https://github.com/langchain-ai/open_deep_research) use a graph with explicit nodes/edges: a supervisor node fans out to parallel "researcher" nodes, each an independent subgraph with its own message history, which then fan back in to a compression/writer node [primary, for what the repo does — no vendor performance claims involved]. Academic precedents predate the Anthropic post: Stanford's STORM system (2024) uses a related but distinct multi-perspective pattern — simulated multi-role conversations (a "writer" persona interviewing a "topic expert" persona from several angles) to build a research outline before drafting — which is a fan-out/fan-in structure applied to perspective diversity rather than parallel search coverage (https://storm-project.stanford.edu/research/storm/) [primary/academic].

## Evidence of maturity and activity

This is a production pattern, not a research curiosity, as of mid-2026. Anthropic has run it in production behind Claude's Research feature since at least June 2025 [primary, vendor self-report for the following performance claims: on Anthropic's internal evaluation, a multi-agent system pairing a Claude Opus 4 lead with Claude Sonnet 4 subagents outperformed a single-agent Claude Opus 4 baseline by 90.2% on a breadth-first research eval; Anthropic also reports that three factors — token usage, tool-call count, and model choice — jointly explain ~95% of variance in performance on the BrowseComp evaluation, with token usage alone explaining ~80%]. These are Anthropic's own numbers from their own eval harness, not independently replicated, and should be read as vendor-reported.

Independent, non-vendor evidence that the pattern generalizes: a Google Research post (Kim & Liu, Jan 28, 2026, arXiv:2512.08296) systematically compares multi-agent vs. single-agent architectures across task types and finds a task-dependent split: on a parallelizable financial-reasoning benchmark, centralized multi-agent coordination beat single-agent by roughly 81%, while on a sequential planning benchmark (PlanCraft), every multi-agent variant tested *underperformed* the single agent by 39–70% [secondary, independent academic]. This is an important cross-check: it corroborates Anthropic's "breadth-first parallel research wins, sequential/tightly-coupled work loses" story with numbers from a different lab and different benchmarks, rather than merely repeating Anthropic's claim. The same paper reports that independent (uncoordinated) multi-agent systems amplify errors ~17.2x versus ~4.4x for centralized (orchestrator-led) ones — direct evidence that the orchestrator-worker topology specifically (versus looser multi-agent designs) matters for reliability.

On adoption: LangChain's open-source `open_deep_research` has accumulated roughly 12,000 GitHub stars and ranked #6 on the independent DeepResearch Bench leaderboard as of an August 2025 update (RACE score 0.4344 default, 0.4943 with a GPT-5 configuration) — DeepResearch Bench itself is an academic benchmark (Du et al., June 16 2025, arXiv:2506.11763) [primary for the repo's own claims; benchmark ranking is third-party]. Commercial "Deep Research" products from OpenAI and Google Gemini are, per multiple secondary write-ups, architected closer to a single strong reasoning-model agent rather than an explicit orchestrator-worker fan-out (https://blog.bytebytego.com/p/how-openai-gemini-and-claude-use) [secondary, unverified against each vendor's own architecture docs — treat as a plausible but not confirmed characterization] — which makes Anthropic's Research feature something of an outlier among the major hosted products in actually being multi-agent under the hood, and underscores why this slice of the report treats "orchestrated multi-agent" as architecturally distinct from those hosted tools even where end-user behavior looks similar.

## Trade-offs

**Quality**: The pattern's advantage is specifically on breadth-first, decomposable queries — "compare these five things," "find every X of type Y" — where independent sub-investigations don't need to see each other's intermediate state. It underperforms on tasks requiring tightly shared context or sequential, order-dependent decisions: Cognition AI's widely-discussed counter-post "Don't Build Multi-Agents" (Walden Yan, 2025, https://cognition.com/blog/dont-build-multi-agents) [secondary, vendor self-report from a competing agent company] argues that sub-agents lacking each other's intermediate decisions produce actively conflicting outputs — their illustrative case is two code sub-agents independently and incompatibly reinterpreting a shared visual style. This is corroborated by the Google Research finding above (sequential/planning tasks degrade under multi-agent decomposition), and by Anthropic's own admission that this is why they don't use the same architecture for their coding agent, since code editing is comparatively "write-heavy" and coupled rather than "read-heavy" and parallel.

**Cost**: The multiplier is steep and consistently reported. Anthropic states agents in general use roughly 4x the tokens of a single chat turn, and multi-agent systems specifically use roughly 15x the tokens of a chat interaction (https://www.anthropic.com/engineering/multi-agent-research-system) [primary, vendor self-report — no independent token-count audit was found]. This figure is repeated consistently across secondary coverage (Simon Willison's write-up, LangChain's blog, multiple Medium/Substack summaries) but all trace back to the same Anthropic post rather than being separately measured, so treat it as a single data point, not independently corroborated.

**Transparency/debuggability**: Multi-agent traces are harder to audit than a single linear transcript — Anthropic explicitly flags non-deterministic behavior between runs (small prompt changes cascade unpredictably through parallel agent populations), state/errors compounding across long tool-call chains, and the need for full production tracing of decision patterns (without exposing conversation contents) just to debug the system at all. This is a maturity gap relative to single-agent systems: there isn't yet a standard, widely-adopted tracing/replay tool specific to orchestrator-worker research pipelines, though general agent-observability tooling (LangSmith, etc.) is being adapted to the purpose.

**Coordination failure modes** observed in practice (per Anthropic and corroborated by LangChain's guidance): orchestrators spawning excessive subagents (50+) for trivial queries before prompt-tuning; subagents endlessly searching for sources that don't exist; subagents duplicating each other's work under vague delegation; subagents preferring SEO-optimized content-farm results over authoritative sources (a failure mode LLM-as-judge evaluation missed but human evaluation caught); and, per Cognition, subagents making silently conflicting implicit decisions when their outputs must later be merged.

**Open-source vs. hosted**: Hosted products (Claude Research, and reportedly OpenAI/Gemini Deep Research) hide this machinery entirely — you get a report, not a trace. Open-source orchestrator-worker implementations (LangChain's `open_deep_research`, `deep_agents`, and hand-rolled variants using Claude Code's own Task/subagent tooling) trade convenience for inspectability: you can watch the orchestrator's delegation prompts, see individual subagent transcripts, and tune the scaling heuristics yourself, at the cost of having to build and maintain the plumbing (retries, rainbow deployments for updating live agents, async result collection) that a hosted vendor handles invisibly.

## Getting-going guide

1. **Pick an orchestration substrate.** Three reasonable starting points as of mid-2026: (a) a hand-rolled orchestrator that calls an LLM API directly and dispatches sub-agent calls as parallel async tasks/tool calls — simplest to reason about, no framework lock-in; (b) LangGraph, which gives you an explicit graph with typed state, a supervisor node, and parallel researcher subgraphs, plus a working reference implementation to fork (`langchain-ai/open_deep_research` or the newer `deep_agents` / `deep_research_from_scratch` repos); (c) a coding-agent's native subagent tooling (e.g., Claude Code's Task tool used non-interactively) if you want to prototype the pattern without writing orchestration code at all.
2. **Define the worker contract first.** Before writing the orchestrator's planning prompt, decide what a worker sub-agent receives (objective, allowed tools, output schema, boundaries) and what it must return (a short structured report, not a raw transcript). Anthropic's biggest lesson is that under-specified delegation is the primary failure source — write the worker task template before the orchestrator logic.
3. **Start with a small, fixed fan-out (3–5 workers)** rather than letting the orchestrator freely decide how many subagents to spawn; add dynamic scaling heuristics (1 agent for simple lookups, more for genuinely independent sub-questions) only after you've observed the fixed version behaving sensibly — Anthropic's own early prototype over-spawned (50+ agents on trivial queries) before this was reined in with explicit prompt rules.
4. **Run workers concurrently, not sequentially**, and give each one permission to fire multiple tool calls in parallel within its own turn — this is where most of the wall-clock benefit comes from (reported up to 90% time reduction), not from having more agents per se.
5. **Add a synthesis pass, and keep citation-mapping as a separate final step** from synthesis rather than asking one call to do both — this mirrors Anthropic's `CitationAgent` split and avoids the model trying to write prose and track source attribution simultaneously.
6. **Instrument before optimizing.** Log each subagent's full transcript and the orchestrator's delegation decisions from day one; you will need this to diagnose duplicated work, hallucinated sources, or SEO-farm bias, none of which show up cleanly in an LLM-as-judge score alone — pair automated rubric evaluation with periodic human spot checks.
7. **Evaluate on a small, fixed query set early** (Anthropic started with ~20 test queries) rather than waiting for a large eval suite; watch for the token-cost multiplier (budget for ~10–15x a single chat call) and gate the multi-agent path behind a query-complexity classifier so trivial questions don't pay the multi-agent tax — route simple lookups to a single agent and reserve fan-out for genuinely breadth-first questions.
8. **Treat this as unsuitable for write-heavy or sequential-decision tasks.** If your target task requires shared, evolving context between steps (e.g., anything closer to code editing than to information-gathering), don't force the orchestrator-worker pattern — both Anthropic and Cognition converge on this boundary from opposite starting points.

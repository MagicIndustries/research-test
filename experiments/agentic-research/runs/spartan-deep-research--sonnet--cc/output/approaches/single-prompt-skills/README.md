# Single-Prompt Research Skills

## Definition

A single-prompt research skill is deep-research capability packaged as one instruction set — a `SKILL.md` file, a slash command, a `.cursorrules`-style prompt, or a single system-prompt template — that runs inside an existing coding or chat harness (Claude Code, Cursor, Codex, Gemini CLI, etc.) and drives one continuous agent loop (optionally one forked subagent) through scoping, searching, verifying, synthesizing, and writing. It is distinguished from multi-agent orchestration by having exactly one reasoning context doing the work end to end, rather than a lead agent spawning a fan-out of parallel subagent researchers with independent context windows. The pattern rides on top of Anthropic's open Agent Skills format (a folder with a `SKILL.md` plus optional scripts/references), published as a cross-vendor standard at agentskills.io on December 18, 2025, and now supported by roughly 40 agent products including Claude Code, Cursor, GitHub Copilot, VS Code, Gemini CLI, OpenCode, and Goose (https://agentskills.io, https://agentman.ai/blog/agent-skills-ecosystem-report-2026).

## How it works

Structurally, a single-prompt research skill is a `SKILL.md` (YAML frontmatter with at minimum `name` and `description`, plus a Markdown body of instructions) that the host agent loads via **progressive disclosure**: at startup only the name/description sit in context; when a task matches, the full body loads; bundled scripts, reference docs, or templates load only as needed (https://agentskills.io, https://code.claude.com/docs/en/skills). This keeps many skills resident without bloating the context window, and is the main mechanical difference from a plain system prompt or `.cursorrules` file, which is always in context.

Within the skill body, the research loop that recurring community implementations converge on looks like:

1. **Scope** — clarify the question, define what "done" looks like, sometimes with a human checkpoint before proceeding (e.g., `/research-add-items` and `/research-add-fields` commands in Weizhena/Deep-Research-skills let a user expand the outline before the agent starts searching: https://github.com/Weizhena/Deep-Research-skills).
2. **Search** — an iterative, ReAct-style think→search→read→think cycle (interleaving reasoning traces with tool calls, per Yao et al.'s original ReAct pattern, arXiv:2210.03629) using the host's built-in `WebSearch`/`WebFetch` tools or an attached MCP search connector (e.g., Exa MCP). One implementation explicitly frames this as searching "deep, not wide — each round follows leads from the last" (https://github.com/recomby-ai/researcher-skill).
3. **Verify/triangulate** — cross-checking claims against multiple sources, sometimes as an explicit pipeline stage ("Triangulate" in the 199-biotechnologies 8-phase pipeline: Scope → Plan → Retrieve → Triangulate → Outline Refinement → Synthesize → Critique → Refine → Package: https://github.com/199-biotechnologies/claude-deep-research-skill).
4. **Critique/self-reflect** — some skills add an explicit self-critique-with-loop-back step, structurally similar to the Reflexion pattern of verbal self-reflection stored in an episodic buffer to improve the next pass (Shinn et al., arXiv:2303.11366).
5. **Synthesize and write** — consolidation into a cited report, often in multiple output formats (Markdown/HTML/PDF).

Tools are typically whatever the host harness already exposes (web search/fetch, file read/write) plus, optionally, a small helper script bundled in the skill folder (for citation-registry bookkeeping, PDF rendering, or job polling against an external research backend, as in NVIDIA's AI-Q skill: https://developer.nvidia.com/blog/add-a-specialized-deep-research-skill-to-agent-harnesses/). Claude Code additionally lets a skill declare `context: fork` / `agent: <subagent-type>` to run in an isolated forked subagent — but this is still one research worker with one context, not a swarm; it differs from Anthropic's own "Research" product architecture, where a lead agent decomposes a query and spawns multiple parallel subagents each with independent context windows and tool-call budgets (https://www.anthropic.com/engineering/multi-agent-research-system). That orchestrator-worker pattern is the multi-agent category this slice explicitly excludes.

## Evidence of maturity and activity

The underlying mechanism (Agent Skills) is young but moving fast: Anthropic's own description of Agent Skills as a production feature dates to October 16, 2025 (https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills, vendor self-report), and the open specification followed two months later, December 18, 2025 (https://agentskills.io). Anthropic's reference repository, anthropics/skills, has 162k stars and 19.2k forks (https://github.com/anthropics/skills) — evidence of broad developer interest in the skills mechanism generally, though the repo's own bundled examples skew toward document/creative tasks rather than a canonical general-purpose research skill.

For research specifically, a visible community layer has formed on top of the spec:
- **199-biotechnologies/claude-deep-research-skill** — 939 stars, 98 forks, MIT, last commit March 19, 2026; an 8-phase pipeline with four depth modes (Quick/Standard/Deep/UltraDeep) (https://github.com/199-biotechnologies/claude-deep-research-skill). Its README claims it "outperforms OpenAI, Gemini, and Claude Desktop" — an unverified, individual-developer self-report with no cited benchmark; treat as marketing, not evidence.
- **Weizhena/Deep-Research-skills** — 1.7k stars, 135 forks, MIT, cross-platform (Claude Code, OpenCode, Codex) (https://github.com/Weizhena/Deep-Research-skills).
- **daymade/claude-code-skills** — 1.3k stars, 211 forks, MIT, 60+ skills including a deep-research skill with source governance and citation tracking (https://github.com/daymade/claude-code-skills).
- **alirezarezvani/claude-skills** — 22.7k stars, 3.1k forks, a 345-skill marketplace-style collection including dedicated academic-research and research-operations categories, last updated May 28, 2026 (https://github.com/alirezarezvani/claude-skills).
- **recomby-ai/researcher-skill** — a much smaller, single-file ReAct-pattern skill (28 stars, 3 forks) that illustrates the minimal end of the spectrum (https://github.com/recomby-ai/researcher-skill).

Adoption of the underlying spec is broad but shallow to measure precisely: the agentskills.io client showcase lists ~40 compatible products as of June 2026, and the SkillsMP directory reports somewhere between 1.9 and 2.2 million indexed `SKILL.md` files depending on the source consulted (https://skillsmp.com/, https://agentman.ai/blog/agent-skills-ecosystem-report-2026) — I couldn't fully reconcile the exact count, and flag it as approximate. No source gave adoption numbers specific to *research* skills as a sub-category (as opposed to skills generally); the star/fork counts above are the best available proxy.

## Trade-offs

**Quality.** Single-prompt skills are well suited to depth-first, boundedly-scoped tasks — a literature review on one topic, a due-diligence pass on one company, a comparison table across a fixed list of items — where following one thread deeply outperforms breadth. Anthropic's own findings from building their multi-agent Research product suggest the opposite regime — "breadth-first queries that involve pursuing multiple independent directions simultaneously" — favors parallel multi-agent decomposition, and that a multi-agent system beat a single Opus agent by 90.2% on their internal eval (https://www.anthropic.com/engineering/multi-agent-research-system, vendor self-report). That gap is contested by newer academic work: Tran & Kiela (arXiv:2604.02460, submitted April 2026) argue that under an *equal thinking-token budget*, single-agent systems match or beat multi-agent systems on multi-hop reasoning, and that previously reported multi-agent advantages are largely an artifact of uncontrolled extra compute rather than architecture — a genuine, currently-unresolved disagreement worth flagging rather than picking a side. Separately, output quality is highly dependent on skill authoring quality: SkillsBench (arXiv:2602.12670), evaluating 47,150 public skills, found an ecosystem-wide average quality score of 6.2/12, and that only curated, top-quartile skills (score ≥9) meaningfully improved pass rates (33.9% → 50.5%, a 25.5% normalized gain) — meaning "install a random research skill" is a much weaker bet than "use a well-maintained one."

**Cost.** Because a single-prompt skill runs in one context (or one forked subagent) rather than fanning out, its token profile is much closer to a normal chat turn than to Anthropic's own multi-agent system, which Anthropic reports uses roughly 15x the tokens of a chat interaction, with token volume alone explaining about 80% of performance variance across their internal evals (https://www.anthropic.com/engineering/multi-agent-research-system). This makes single-prompt skills the cheaper default and the right choice unless the task's value clearly justifies the multi-agent premium. The trade is wall-clock time and ceiling on breadth: a lone context can't truly parallelize independent search branches, so genuinely multi-pronged investigations may take longer or require more turns to reach the same coverage.

**Transparency.** A single linear transcript (one agent, one running context) is materially easier to audit step-by-step than reconstructing which of several parallel subagents surfaced which claim — there's no cross-agent handoff to trace. This is a real advantage for compliance-sensitive or reviewable research work, though it depends entirely on the skill actually showing its reasoning and citations rather than collapsing straight to a final answer.

**Open-source vs. hosted.** Because Agent Skills is now a published, vendor-neutral spec, a research skill written once is portable across Claude Code, Cursor, Copilot, Gemini CLI, and more — unlike a hosted "Research" product, which is closed, run entirely on the vendor's infrastructure, and architecturally multi-agent by default. The trade-off is supply-chain risk: installing a third-party skill means running unreviewed instructions/code from someone else's repo. Snyk's ToxicSkills audit of 3,984 skills sourced from ClawHub and skills.sh (as of February 5, 2026) found 36% contained security flaws, 13.4% (534) contained at least one critical issue (malware, prompt injection, exposed secrets), and confirmed 76 malicious payloads, 8 still live at publication (https://snyk.io/blog/toxicskills-malicious-ai-agent-skills-clawhub/). A separate, larger figure circulating in secondary coverage — "22,511 skills audited, 140,963 issues" — is attributed by Agentman's blog to "Agensi/Snyk, 2026" but the primary source could not be located and the number does not match Snyk's own 3,984-skill study directly (https://agentman.ai/blog/agent-skills-ecosystem-report-2026); treat this larger figure as unverified pending a primary source.

## Getting-going guide

1. Pick a host harness that supports the Agent Skills spec — Claude Code is the most documented starting point (https://code.claude.com/docs/en/skills); Cursor, Copilot, and Gemini CLI also support the same format (https://agentskills.io).
2. Create a skill folder, e.g. `.claude/skills/deep-research/SKILL.md` (project-scoped) or `~/.claude/skills/deep-research/SKILL.md` (user-scoped), with YAML frontmatter declaring at minimum `name` and `description` — the description is what triggers automatic activation, so make it specific about when the skill should fire.
3. Write the loop into the body: scope/clarify → iterative ReAct-style search → cross-source verification → optional critique/loop-back → synthesis with citations → write-up. Use one of the public repos above as a concrete starting template rather than writing from scratch: `recomby-ai/researcher-skill` for a minimal single-file version, `Weizhena/Deep-Research-skills` for a human-in-the-loop outline-then-research flow, or `199-biotechnologies/claude-deep-research-skill` for a fuller 8-phase pipeline with a `reference/` directory pattern that keeps the top-level `SKILL.md` lean (~100 lines) while pushing methodology detail into files loaded on demand.
4. Decide search tooling: the host's built-in web search/fetch is enough to start; for higher-recall search, wire in an MCP search connector (Weizhena's OpenCode variant uses `OPENCODE_ENABLE_EXA=1` for Exa MCP).
5. Keep the skill's own footprint small and push detail into `references/`, `scripts/`, and `assets/` subfolders — this is what makes progressive disclosure actually work rather than just moving the token cost around.
6. If the research is long or you don't want it to pollute your main conversation's context, use Claude Code's subagent execution option (`context: fork` in frontmatter) — this still runs as one research worker, not a swarm, keeping it in this category.
7. Test against real queries and watch specifically for two common failure modes: the skill not activating because the description is too vague, and the agent asserting claims without the verification step actually running. SkillsBench's quality gap (6.2/12 average vs. top-quartile skills) suggests grading your own skill against a rubric before relying on it is worthwhile (https://arxiv.org/abs/2602.12670).
8. If you plan to install third-party research skills rather than write your own, treat them as a supply-chain dependency — check the repo, read the `SKILL.md` before installing, and be aware of the ToxicSkills findings above before adopting community skills wholesale.
